Summer Research Project Ideas

These ideas are meant for internal circulation only. Please do not post!

Below are several ideas I have for summer research projects. These are all implementation-oriented, as I think a hands-on project works best for the summer format. Does one interest you? Let's talk!

Official program info

  • Merging GHC's term-level and type-level parsers. Part of the work to add dependent types to Haskell is to take GHC's current term-level and type-level parsers and merge them together, into one parser to rule them all. This work would be a concrete step toward building dependent types in a practical language and would give you a great deal of experience working in a real-world programming language compiler.

  • Improving GHC's error messages. Currently, type errors are produced at the convenience of GHC. These messages are static and very old-fashioned: there are no colors, no interactivity, and no good way for them to interact with an IDE. In addition, they are sometimes awfully hard to understand.

    This project would be to rewrite GHC's error-message-handling mechanism, with two goals:

    1. Make error messages structured, allowing for interactivity (e.g., hiding parts of the message), fancy formatting, and IDE integration
    2. Improve the subjective quality of error messages, making use of the fancy features you established in goal (1).
    There is a good deal of interest in getting this done in the Haskell community; successful work here would gain you a lot of love.

    Some background material:

    • A talk on how Idris structures error messages, which I'd like to emulate.
    • A talk on type provenance a technique which might make error messages more understandable.
    • Though much more theoretical, this abstract describes a potential way to re-imagine type errors to make them more systematic.

  • Implementing GHC proposals. In the course of my main research thrust of adding dependent types to GHC, I have made a number of proposals through GHC's official proposal process. This process is hosted on GitHub. This list of active proposals includes several of mine (under username `goldfirere`), any of which would make a suitable part (or whole) of a summer project.

  • Visible type patterns. Some recent work has introduced type applications to Haskell. These type applications can be used only in expressions, never in patterns. Users are thus asking to extend the new feature to patterns, as tracked in this ticket and this proposal. (Do also see this comment, which discusses some of the challenges.) This project would be to specify and implement the new feature, hopefully leading to a Haskell Symposium publication.

  • Mechanized proofs of Haskell libraries. An effort is underway to formally prove the correctness of various aspects of the functional programming ecosystem. A recent paper presents a tool for translating Haskell code to Coq code. Coq is a proof assistant that allows you to write a mathematically valid proof in computer code; your proof is checked by the Coq compiler.

    The summer work would be to take certain Haskell libraries, run them through the hs-to-coq translator, and then write the proofs of correctness. This would be a great first experience in verified programming, and I'm confident you would find bugs in real, published libraries that could then be corrected.

  • Improving GHC's pattern-match warnings. GHC currently helpfully reports when a pattern is redundant, unreachable, or incomplete. This warning is essential for writing error-free code. (Some languages make it an error to write an incomplete pattern-match.) However, the check has many deficiencies. This project would be to work through this area of GHC and try to make this check better. This could be a great first experience in open-source development and would be a real contribution to the GHC community. Required background reading: the paper that describes the pattern-match checker.
